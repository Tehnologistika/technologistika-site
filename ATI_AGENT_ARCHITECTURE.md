# Архитектура ATI-агента ТехноЛогистики

## 1. Обзор

Данный документ описывает архитектуру AI-агента, предназначенного для автоматизации взаимодействия с платформой ATI.su и мессенджером MAX. Целью агента является оптимизация логистических процессов компании «ТехноЛогистика» путем автоматизации публикации заявок, поиска перевозчиков, ведения переговоров и анализа рыночных ставок.

## 2. Принципы проектирования

*   **Безопасность:** Приоритет отдается безопасным методам интеграции, минимизации рисков несанкционированного доступа и автоматических действий без подтверждения.
*   **Модульность:** Архитектура должна быть модульной для облегчения разработки, тестирования и масштабирования.
*   **Расширяемость:** Возможность добавления новых функций и интеграций в будущем.
*   **Надежность:** Обеспечение стабильной работы и обработки ошибок.
*   **Контролируемость:** Все автоматические действия должны быть подтверждены вручную на начальном этапе.

## 3. Общая архитектура

ATI-агент будет представлять собой микросервисную архитектуру, состоящую из нескольких ключевых компонентов, взаимодействующих друг с другом и с внешними системами.

```mermaid
graph TD
    A[MAX Messenger] -- Заявки --> B(MAX Messenger API Handler)
    B -- Обработанные заявки --> C(Core Logic / Orchestrator)
    C -- Публикация / Поиск / Сообщения --> D(ATI.su API Handler)
    D -- Взаимодействие с ATI.su --> E[ATI.su Platform]
    C -- Управление данными --> F(Database)
    C -- Аналитика --> G(Market Rate Analyzer)
    G -- Данные о ставках --> D
    C -- Уведомления / Подтверждения --> H[Gmail / User Interface]
    H -- Взаимодействие с пользователем --> C
    F -- Хранение данных --> I[Google Sheets (MVP)]
    I -- Синхронизация --> F
```

## 4. Компоненты системы

### 4.1. MAX Messenger API Handler

*   **Назначение:** Получение заявок из MAX Messenger.
*   **Технологии:** Использование официального API MAX Messenger для ботов.
*   **Функционал:**
    *   Подключение к MAX Messenger API.
    *   Парсинг входящих сообщений/заявок.
    *   Передача структурированных данных в Core Logic.

### 4.2. ATI.su API Handler

*   **Назначение:** Взаимодействие с платформой ATI.su через официальные API.
*   **Технологии:** Использование ATI.su Loads API, Trucks Search API, Messenger API, Average Rates API, Price Prediction API.
*   **Функционал:**
    *   **Публикация заявок:** Создание и обновление грузов на ATI.su.
    *   **Поиск перевозчиков:** Использование Trucks Search API для поиска подходящих транспортных средств.
    *   **Отправка сообщений:** Взаимодействие с перевозчиками через ATI.su Messenger API.
    *   **Получение ставок:** Сбор предложений от перевозчиков.
    *   **Получение рыночных данных:** Использование Average Rates API и Price Prediction API для анализа цен.
*   **Безопасность:** Реализация механизмов аутентификации и авторизации для доступа к ATI.su API. В случае отсутствия API для критически важных функций, предусмотрен вариант с Browser Automation (Playwright) с обязательным ручным подтверждением.

### 4.3. Core Logic / Orchestrator

*   **Назначение:** Центральный компонент, управляющий бизнес-логикой агента, координирующий взаимодействие между другими компонентами.
*   **Функционал:**
    *   **Обработка заявок:** Прием заявок из MAX Messenger, их валидация и подготовка к публикации на ATI.su.
    *   **Управление рабочими процессами:** Оркестрация сценариев (публикация, поиск, переговоры).
    *   **Принятие решений:** На основе данных и правил, принятие решений о дальнейших действиях (например, кому отправить сообщение, какую ставку предложить).
    *   **Взаимодействие с пользователем:** Отправка уведомлений и запросов на подтверждение действий через Gmail или специализированный UI.
    *   **Аудит:** Запись всех действий и решений в Audit Log.

### 4.4. Database

*   **Назначение:** Хранение всех операционных данных агента.
*   **Технологии:** Реляционная база данных (например, PostgreSQL).
*   **Функционал:** Хранение информации о заявках, перевозчиках, ставках, переговорах, сообщениях, статусах одобрения и логах аудита.

### 4.5. Market Rate Analyzer

*   **Назначение:** Анализ рыночных ставок и расчет актуальной рыночной стоимости по маршрутам.
*   **Функционал:**
    *   Получение данных из ATI.su Average Rates API и Price Prediction API.
    *   Обработка и агрегация данных.
    *   Расчет оптимальных ставок и прогнозирование цен.
    *   Предоставление аналитических данных Core Logic.

### 4.6. Gmail / User Interface

*   **Назначение:** Канал взаимодействия с пользователем для уведомлений, запросов на подтверждение и ручного управления.
*   **Технологии:** Gmail API для отправки/получения писем, возможно, простой веб-интерфейс для отображения статусов и запросов на подтверждение.
*   **Функционал:**
    *   Отправка уведомлений о новых заявках, предложениях, требующих внимания.
    *   Получение подтверждений от пользователя для выполнения критических действий (например, публикация груза, отправка сообщения).
    *   Отображение текущего статуса операций.

## 5. Схема базы данных

Предлагаемая структура базы данных включает следующие таблицы:

### `source_requests` (Исходные заявки)

*   `id` (UUID, Primary Key): Уникальный идентификатор заявки.
*   `source_system` (VARCHAR): Система-источник заявки (например, 'MAX Messenger').
*   `source_id` (VARCHAR): Идентификатор заявки в исходной системе.
*   `payload` (JSONB): Полный JSON-объект исходной заявки.
*   `status` (VARCHAR): Статус обработки заявки (например, 'new', 'processed', 'error').
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `ati_posts` (Публикации на ATI.su)

*   `id` (UUID, Primary Key): Уникальный идентификатор публикации.
*   `source_request_id` (UUID, Foreign Key to `source_requests.id`): Ссылка на исходную заявку.
*   `ati_load_id` (VARCHAR): Идентификатор груза на ATI.su.
*   `status` (VARCHAR): Статус публикации (например, 'draft', 'published', 'archived', 'error').
*   `payload` (JSONB): JSON-объект данных, отправленных в ATI.su API для публикации.
*   `response` (JSONB): JSON-объект ответа от ATI.su API.
*   `published_at` (TIMESTAMP): Дата и время публикации.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `carriers` (Перевозчики)

*   `id` (UUID, Primary Key): Уникальный идентификатор перевозчика.
*   `ati_id` (VARCHAR, Unique): Идентификатор перевозчика на ATI.su.
*   `name` (VARCHAR): Название компании перевозчика.
*   `contact_person` (VARCHAR): Контактное лицо.
*   `phone` (VARCHAR): Телефон.
*   `email` (VARCHAR): Email.
*   `rating` (DECIMAL): Рейтинг перевозчика на ATI.su.
*   `is_verified` (BOOLEAN): Флаг верификации.
*   `notes` (TEXT): Дополнительные заметки.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `carrier_contacts` (Контакты перевозчиков)

*   `id` (UUID, Primary Key): Уникальный идентификатор контакта.
*   `carrier_id` (UUID, Foreign Key to `carriers.id`): Ссылка на перевозчика.
*   `type` (VARCHAR): Тип контакта (например, 'phone', 'email', 'ati_messenger').
*   `value` (VARCHAR): Значение контакта.
*   `description` (TEXT): Описание контакта.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `routes` (Маршруты)

*   `id` (UUID, Primary Key): Уникальный идентификатор маршрута.
*   `origin_city_id` (INTEGER): ID города отправления (из геословаря ATI.su).
*   `origin_city_name` (VARCHAR): Название города отправления.
*   `destination_city_id` (INTEGER): ID города назначения (из геословаря ATI.su).
*   `destination_city_name` (VARCHAR): Название города назначения.
*   `distance_km` (INTEGER): Расстояние в километрах.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `rates` (Ставки)

*   `id` (UUID, Primary Key): Уникальный идентификатор ставки.
*   `ati_post_id` (UUID, Foreign Key to `ati_posts.id`): Ссылка на публикацию груза.
*   `carrier_id` (UUID, Foreign Key to `carriers.id`): Ссылка на перевозчика, предложившего ставку.
*   `amount` (DECIMAL): Предложенная сумма ставки.
*   `currency` (VARCHAR): Валюта ставки.
*   `payment_type` (VARCHAR): Тип оплаты (например, 'cash', 'with_nds', 'without_nds').
*   `is_counter_offer` (BOOLEAN): Флаг встречного предложения.
*   `received_at` (TIMESTAMP): Дата и время получения ставки.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `negotiations` (Переговоры)

*   `id` (UUID, Primary Key): Уникальный идентификатор переговоров.
*   `ati_post_id` (UUID, Foreign Key to `ati_posts.id`): Ссылка на публикацию груза.
*   `carrier_id` (UUID, Foreign Key to `carriers.id`): Ссылка на перевозчика.
*   `status` (VARCHAR): Статус переговоров (например, 'initiated', 'in_progress', 'agreed', 'rejected').
*   `start_time` (TIMESTAMP): Время начала переговоров.
*   `end_time` (TIMESTAMP): Время окончания переговоров.
*   `final_rate_id` (UUID, Foreign Key to `rates.id`, Nullable): Ссылка на финальную ставку.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `messages` (Сообщения)

*   `id` (UUID, Primary Key): Уникальный идентификатор сообщения.
*   `negotiation_id` (UUID, Foreign Key to `negotiations.id`): Ссылка на переговоры.
*   `sender_type` (VARCHAR): Тип отправителя ('agent', 'carrier', 'user').
*   `sender_id` (VARCHAR): Идентификатор отправителя (например, ID агента, ID перевозчика, ID пользователя).
*   `recipient_type` (VARCHAR): Тип получателя ('agent', 'carrier', 'user').
*   `recipient_id` (VARCHAR): Идентификатор получателя.
*   `content` (TEXT): Текст сообщения.
*   `sent_at` (TIMESTAMP): Дата и время отправки сообщения.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `approvals` (Одобрения)

*   `id` (UUID, Primary Key): Уникальный идентификатор одобрения.
*   `entity_type` (VARCHAR): Тип сущности, требующей одобрения (например, 'ati_post', 'message_send', 'rate_accept').
*   `entity_id` (UUID): Идентификатор сущности.
*   `requested_by` (VARCHAR): Кто запросил одобрение (например, 'agent').
*   `approved_by` (VARCHAR): Кто одобрил (например, 'user').
*   `status` (VARCHAR): Статус одобрения ('pending', 'approved', 'rejected').
*   `request_payload` (JSONB): Данные, запрошенные для одобрения.
*   `response_payload` (JSONB): Ответ пользователя.
*   `requested_at` (TIMESTAMP): Дата и время запроса.
*   `approved_at` (TIMESTAMP): Дата и время одобрения.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

### `audit_log` (Журнал аудита)

*   `id` (UUID, Primary Key): Уникальный идентификатор записи аудита.
*   `timestamp` (TIMESTAMP): Дата и время события.
*   `event_type` (VARCHAR): Тип события (например, 'load_published', 'carrier_searched', 'message_sent', 'approval_requested').
*   `entity_type` (VARCHAR): Тип сущности, к которой относится событие.
*   `entity_id` (UUID): Идентификатор сущности.
*   `user_id` (VARCHAR, Nullable): Идентификатор пользователя, инициировавшего действие (если применимо).
*   `details` (JSONB): Дополнительные детали события.
*   `created_at` (TIMESTAMP): Дата и время создания записи.
*   `updated_at` (TIMESTAMP): Дата и время последнего обновления записи.

## 6. Сценарии работы

### 6.1. Заявка из MAX → Черновик ATI

1.  **Получение заявки:** MAX Messenger API Handler получает новую заявку из MAX Messenger.
2.  **Парсинг и валидация:** Core Logic парсит заявку, извлекает ключевые параметры (маршрут, тип груза, вес, объем, даты) и валидирует их.
3.  **Сохранение исходной заявки:** Заявка сохраняется в таблице `source_requests` со статусом 'new'.
4.  **Подготовка черновика:** Core Logic формирует черновик груза для ATI.su на основе данных заявки.
5.  **Запрос на одобрение:** Core Logic отправляет запрос пользователю (через Gmail/UI) на одобрение публикации черновика на ATI.su. Запись создается в таблице `approvals`.
6.  **Ожидание одобрения:** Агент ожидает подтверждения от пользователя.
7.  **Публикация (после одобрения):** После получения одобрения, ATI.su API Handler использует Loads API для создания груза на ATI.su. Запись создается в таблице `ati_posts` со статусом 'draft' или 'pending_publication', затем обновляется на 'published' после успешной публикации.
8.  **Логирование:** Все шаги логируются в `audit_log`.

### 6.2. Поиск перевозчиков

1.  **Инициация поиска:** Core Logic инициирует поиск перевозчиков для опубликованного груза или по запросу пользователя.
2.  **Параметры поиска:** Core Logic формирует параметры для поиска на основе данных груза (маршрут, тип транспорта, даты).
3.  **Запрос к ATI.su API:** ATI.su API Handler использует Trucks Search API для поиска подходящих перевозчиков.
4.  **Обработка результатов:** Core Logic получает список потенциальных перевозчиков. Данные о перевозчиках сохраняются или обновляются в таблице `carriers` и `carrier_contacts`.
5.  **Фильтрация и ранжирование:** Core Logic фильтрует и ранжирует перевозчиков на основе заданных критериев (рейтинг, история сотрудничества, наличие контактов).
6.  **Логирование:** Все шаги логируются в `audit_log`.

### 6.3. Письмо перевозчику

1.  **Выбор перевозчика:** Core Logic выбирает перевозчика из списка (возможно, с ручным подтверждением пользователя).
2.  **Подготовка сообщения:** Core Logic генерирует черновик сообщения для перевозчика, включающий детали груза и запрос ставки.
3.  **Запрос на одобрение:** Core Logic отправляет запрос пользователю на одобрение отправки сообщения. Запись создается в таблице `approvals`.
4.  **Отправка сообщения (после одобрения):** После получения одобрения, ATI.su API Handler использует Messenger API для отправки сообщения перевозчику. Сообщение сохраняется в таблице `messages`.
5.  **Логирование:** Все шаги логируются в `audit_log`.

### 6.4. Получение ставки

1.  **Мониторинг сообщений:** ATI.su API Handler постоянно мониторит входящие сообщения в ATI.su Messenger API.
2.  **Парсинг ставки:** При получении сообщения, содержащего предложение ставки, Core Logic парсит его и извлекает сумму, валюту и другие детали.
3.  **Сохранение ставки:** Ставка сохраняется в таблице `rates`.
4.  **Уведомление пользователя:** Core Logic уведомляет пользователя о новой ставке (через Gmail/UI).
5.  **Логирование:** Все шаги логируются в `audit_log`.

### 6.5. Обновление таблицы перевозчиков и цен

1.  **Периодическое обновление:** Core Logic периодически запускает процесс обновления данных о перевозчиках и ценах.
2.  **Сбор данных:** ATI.su API Handler собирает актуальные данные о перевозчиках (рейтинги, контакты) и рыночных ставках (Average Rates API, Price Prediction API).
3.  **Обновление БД:** Core Logic обновляет таблицы `carriers`, `carrier_contacts`, `rates` и `routes` на основе полученных данных.
4.  **Логирование:** Все шаги логируются в `audit_log`.

### 6.6. Расчет рыночной цены

1.  **Запрос на расчет:** Core Logic получает запрос на расчет рыночной цены для конкретного маршрута и типа груза (либо от пользователя, либо автоматически).
2.  **Сбор данных:** Market Rate Analyzer использует ATI.su Average Rates API и Price Prediction API для получения исторических и текущих данных по заданным параметрам.
3.  **Анализ и расчет:** Market Rate Analyzer обрабатывает данные, применяет алгоритмы для расчета актуальной рыночной стоимости и ее диапазона.
4.  **Предоставление результата:** Core Logic предоставляет рассчитанную рыночную цену пользователю (через Gmail/UI) или использует ее для формирования предложений перевозчикам.
5.  **Логирование:** Все шаги логируются в `audit_log`.

## 7. Альтернатива: Browser Automation (Playwright)

В случае, если для какой-либо из требуемых функций отсутствует официальный API или его использование нецелесообразно, будет рассмотрен вариант с Browser Automation через Playwright.

*   **Принцип работы:** Playwright будет имитировать действия пользователя в браузере (вход на сайт, навигация, заполнение форм, клики).
*   **Безопасность:** **Крайне важно:** Все действия, выполняемые через Browser Automation, будут требовать **ручного подтверждения** пользователя перед выполнением. Это исключает несанкционированные или ошибочные автоматические действия.
*   **Сценарии:** Может быть использован для публикации заявок, если API не предоставляет нужного функционала, или для сбора данных, недоступных через API.
*   **Риски:** Большая хрупкость по сравнению с API (изменения в UI сайта могут сломать автоматизацию), потенциальные проблемы с обнаружением ботов со стороны ATI.su.

## 8. Интеграция с Google Workspace

*   **Google Sheets:** Будет использоваться для MVP в качестве хранилища данных (см. `ATI_AGENT_MVP_PLAN.md`). Core Logic будет синхронизировать данные с Google Sheets.
*   **Gmail:** Используется для отправки уведомлений и получения подтверждений от пользователя.
*   **Google Drive:** Может быть использован для хранения отчетов или других файлов, генерируемых агентом.

## 9. Развертывание и масштабирование

На начальном этапе агент будет развернут в изолированной среде. В дальнейшем, при необходимости, архитектура позволит масштабировать отдельные компоненты (например, несколько экземпляров ATI.su API Handler для параллельной обработки запросов) и развертывать их в контейнеризированной среде (Docker, Kubernetes).

## 10. Безопасность и аудит

*   **Аутентификация/Авторизация:** Использование токенов API для доступа к внешним сервисам. Управление доступом к самому агенту.
*   **Логирование:** Все действия агента, включая запросы к API, ответы, решения и подтверждения пользователя, будут записываться в `audit_log` для обеспечения прозрачности и возможности отслеживания.
*   **Ручное подтверждение:** На первом этапе все критические действия требуют ручного подтверждения пользователя.

## 11. Дальнейшее развитие

*   Разработка специализированного пользовательского интерфейса для управления агентом.
*   Интеграция с другими логистическими платформами.
*   Внедрение более сложных моделей машинного обучения для оптимизации ставок и выбора перевозчиков.
